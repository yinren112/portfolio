## 为什么做这个

每天花两个多小时在 BOSS 直聘上滚岗位列表、点开看 JD、判断靠不靠谱、复制粘贴到 AI 帮忙看——这事我干了快两周之后觉得不对劲：流程是机械的，判断标准是固定的，这应该让程序来做。

但市面上的自动投递工具有一个问题——它们替你按"立即沟通"按钮，一旦误触了不该投的公司（培训贷、挂羊头卖狗肉的销售岗），你连撤回的机会都没有。

所以我没做"自动投递"，做的是把"抓取→筛选→人工确认"这条流水线搭起来——**最终决定投不投的权限，始终留在人手里**。

**配套的 JD 收集器插件**
单独写了一个 Chrome 扩展（boss-jd-grabber），在 BOSS 直聘列表页自动遍历岗位卡片、读取 JD、清洗成结构化数据导出。遍历时模拟真人节奏做请求节流，数据全存在本地 IndexedDB，不上传到任何第三方服务。

## 这是一个 Agent 原生设计的系统

【组件：小标题+强调框，视觉上跟正文区分开】

"写了个脚本、顺便调了下AI接口"不足以描述这套系统——它从一开始就是按Agent的方式设计的：**感知**（抓取岗位数据）→**推理**（规则先筛一轮，LLM 再判断岗位和自身经历的匹配度）→**行动**（生成结构化日报、给出投递建议）→**观察**（结果反馈进下一轮的去重库和公司黑名单，让判断越来越准）。

唯一刻意收窄的是"行动"这一步的自主权限——最终点"沟通"按钮的权限始终留在人手里，不让Agent自主完成投递。这是一个**主动的Agent设计选择**：任务的风险和后果（投错公司、骚扰HR）不对称地高，所以人机协同的边界划在"决策"和"执行"之间，而不是划在"要不要用AI"这个层面。

## 关键技术决策

### 1. 为什么手写 CDP 客户端，不用 Playwright/Selenium

Playwright 和 Selenium 会启动自己管控的浏览器实例。我的需求是**连接到已经登录好的、带完整 Cookie 和登录态的真实 Chrome 窗口**——三个账号分别跑在三个 Chrome profile 里，每个窗口开不同的调试端口（19222/19223/19224），程序通过 WebSocket 连上去操作。

这个场景用现成框架反而不方便。手写 CDP 客户端（基于 `ws` 库直连 Chrome DevTools Protocol），代码量多了一些，但每个操作都能精确控制，不依赖第三方封装。

### 2. 文件即数据库

没用 MySQL/MongoDB/SQLite，所有数据（岗位、去重记录、日报）都是 JSONL 和 JSON 文件。这是因为：数据量小（每天几百条级别，不是百万级），文件直接用文本编辑器就能看，出问题直接 grep 定位，不需要连数据库才能排查。对这个体量的工具来说，文件系统就是最好的数据库。

### 3. 三层去重

同一个岗位可能出现在三个账号的推荐流里，也可能今天推过明天换个标题再推。去重逻辑走三层：加密岗位 ID → 公司名+岗位名组合 → JD 文本相似度。实测三账号并行 155 条推荐，去重后 141 条唯一，重复率只有 9%。

### 4. 为什么"AI精筛"是一份Prompt模板，不是写死的API调用

岗位终筛没有接 LLM API，而是把判断规则写成 `prompts/02_final_filter_and_draft.md`，每天用当时在开的编码 Agent 照着跑：6 类硬性剔除词（培训贷/刷单、产线伪装、外貌门槛、保险增员、隐藏销售、跑不赢保底线），过审的岗位逐条写 45-120 字定制开场白。Agent 的输出不是照单全收——`validate_approved.js` 会机械校验：`judge.evidence` 必须是 JD 原文里能查到的原句，不是编出来的；开场白长度卡在 35-140 字；同一岗位 ID 不能重复；同一家公司一批最多批准 2 个。判断交给 Agent，兜底交给代码。

## 踩坑：误投事故和三道安全闸门

2026 年 6 月 4 日，自动流程把一条不该投的岗位发了出去。事后排查发现是跨批次数据没做完整去重，导致前一天已经标记为"跳过"的公司在第二天重新出现并被投递。

当天加了三道硬闸门：
1. **跨批次去重**——不只在当天去重，历史所有批次处理过的岗位 ID 全局排除
2. **公司风险名单**——手动维护一份公司黑名单，在名单里的直接跳过不进入 AI 筛选
3. **单公司每日限投**——同一家公司一天最多投递 1 条，就算它同时挂了 5 个岗位

这三道闸门之后再没出过误投。
